# Adversarial Review — Epic 7 Amendments (AD-26, AD-27, AD-4 v1-driver amendment, AD-11 honest-gap additions)

Reviewer lens: construct two units one level down — two Epic 7 stories, or the harness-side skill vs. the core-side `kagami` CLI, or two PR reviewers judging the same story — that each obey every AD to the letter yet build incompatibly, or that would honestly disagree on pass/fail. Scope: the four amendments dated 2026-07-03 layered onto the otherwise-final 2026-07-02 spine. Prior full-spine adversarial pass is `review-adversarial.md`; this pass targets only the new/amended material.

Verdict: **7 holes**. Holes 1–2 are shared-contract gaps that let two Epic 7 implementers build genuinely incompatible systems while each cites AD-26 as their authority. Holes 3 is a security-posture gap masquerading as a settled deferral. Hole 4 is an unfalsifiable acceptance gate. Holes 5–7 are direct textual tensions with unamended, still-`[ADOPTED]` ADs (AD-1, AD-11, AD-8/AD-14) that the spine's own amendment discipline (dated notes on AD-4 and AD-11) was not applied to.

---

## Hole 1 — AD-26(a)'s refusal-counter has no storage location, and AD-20 leaves it no legal home

**The two compliant constructions.** AD-26(a): "The chokepoint tracks consecutive identical refusals per `(entrypoint, target)` under the run lock (AD-15)." AD-15 names `.lock` and `.lease` as the lock/lease *files*, not data stores — "under the run lock" describes serialization, not persistence. Every `kagami` invocation is a separate OS process (established in `review-adversarial.md` Hole 1's still-valid premise: CLI calls are not a long-lived server), so the counter cannot be in-memory; it must live on disk somewhere, and AD-26 never says where.

- **Implementer A** adds `refusal_counts: {entrypoint: target: n}` to `manifest.yaml`, reasoning by analogy to AD-20's "cache plus the non-derivable facts" framing and AD-15's "fixed write order per operation — version file → meta → event."
- **Implementer B** adds a `retries` block to the per-artifact `meta.yaml`, reasoning by analogy to AD-10's claims table living there.

Both are wrong by AD-20's own letter: "The manifest's authoritative contents are **only** what is not derivable: run id, rooting Intuition Note reference, depth budgets, and monitoring config" — a closed, four-item enumeration with no fifth slot for a refusal counter. B's choice fails for any entrypoint whose "target" isn't an artifact with a `meta.yaml` at all — `kagami ask emit`, `kagami scan`, `kagami monitor`, `kagami run open` have no natural per-artifact home. Two Epic 7 stories (one implementing the ceiling for `kagami write`, another for `kagami ask emit`) built independently would put the counter in two different files with two different schemas, and a third story auditing "did the ceiling fire" would not know which file to read.

Compounding gaps in the same clause:
- **"Identical" is undefined.** The key given is `(entrypoint, target)` only — no mention of payload, error code, or refusal reason. Read literally, *any* two consecutive refusals on the same `(entrypoint, target)` count, regardless of why each was refused. But the PRD source (FR-48: "consecutive identical refusals against the same target") and AD-26's own prose ("refusal-retry ceiling") both use "identical," inviting a second, narrower reading: the counter increments only when the *refusal reason* repeats, resetting when the reason changes. These diverge concretely: a harness that hits a window-not-open refusal, then (after legitimately fixing that) a claim-conflict refusal, then a schema-validation refusal — three different reasons, same `(entrypoint, target)` — trips Implementer A's ceiling after 3 attempts but never trips Implementer B's (each reason resets B's count to 1). Neither implementer is off the spec; the spec doesn't say which axis "identical" ranges over.
- **No reset semantics.** Nothing says a successful write resets the counter, or what happens to the count across a researcher's `requires_researcher` resolution — does the count zero out, or does the very next refusal on that target immediately re-trip `requires_researcher` because the stale count is still at ceiling? `kagami run open`'s consistency repair (AD-15) reaps expired claims explicitly; nothing parallel is specified for a refusal counter it can't even locate.
- **"A configured ceiling"** — configured where? AD-12 established the convention that researcher-owned, non-hardcoded values live in `config.yaml`; AD-26 never says the ceiling does. One implementer exposes `refusal_ceiling` in `config.yaml` (consistent with AD-12's convention); another hardcodes a constant in `kagami/kernel/` (also compliant with AD-26's literal silence, but makes "configured" a lie and diverges from AD-12's spirit).

**Close with:** name the storage location explicitly (a new control-plane file, e.g. `runs/<run-id>/.refusals`, or an explicit fifth field added to AD-20's enumerated manifest list — but then AD-20's "only" must be edited too); define "identical" as the `(entrypoint, target, refusal_error_code)` triple with explicit reset-on-success semantics; place the ceiling value in `config.yaml` per AD-12's existing convention and say so.

## Hole 2 — AD-26(b)'s `llm_call` reporting entrypoint is the one new CLI verb in the whole spine left unnamed

**The pattern everywhere else in the document:** every other entrypoint the spine introduces gets a literal name, even ones explicitly deferred: `kagami session open/close` (AD-4, deferred but still named), `kagami scan`, `kagami accept`, `kagami ask emit/answer/revise` (AD-8), `kagami run open` (AD-15), `kagami migrate` (AD-23), `kagami monitor` (AD-24), `kagami metrics` (AD-11/AD-26c). AD-26(b) is the sole exception: "the harness calls **a new validating `kagami` entrypoint**... before the `llm_call` event is appended." The capability map (row for FR-48/FR-49) repeats the same unnamed phrasing: "new reporting entrypoint." FR-49 in the PRD is equally unnamed ("through a validated entrypoint"), so the gap is inherited, not introduced — but resolving exactly this kind of gap is the architecture layer's job, and it is the one place in the document where that job was skipped.

**The incompatibility.** Two Epic 7 stories — one implementing the skill-side call-site (`agents/*.md` frontmatter or the Interviewer skill), one implementing the CLI-side entrypoint — built by two people from AD-26(b) alone would each invent a name (`kagami llm-call report`, `kagami report-call`, `kagami telemetry log`, `kagami event llm-call`) with no shared contract to converge on. Since these are literally two ends of one CLI call, a naming mismatch isn't a style disagreement, it's a broken integration discovered only at run time — exactly the "two units obey every AD, build incompatibly" failure mode this review is looking for, and the cheapest one in the whole set to close (name it, the way every sibling AD already does).

**A second gap in the same clause: no idempotency key.** AD-11's honest-gap register covers *under*-reporting ("a call made but never reported is invisible to the ledger... logged as an AD-11 honest-gap, not silently corrected") but nothing addresses *over*-reporting. A harness retry bug, a crash-and-resume, or a naive "call it defensively twice to be safe" implementation can call the reporting entrypoint twice for one model invocation. AD-26(c)'s soft-limit warning (FR-37) and O6's "chronic-overrun evidence" both consume the cumulative token ledger this entrypoint feeds — double-reporting silently inflates both, and nothing (no call-id, no dedupe window) lets the entrypoint or a later audit detect it. Two implementers again diverge: one adds a client-supplied idempotency token to the entrypoint's arguments, one doesn't, and the resulting event streams are not comparable across the two toy-run protocols AD-27(c) is supposed to make "golden" and repeatable.

**Close with:** name the entrypoint (e.g. `kagami llm-call report`) in AD-26(b) itself, matching the naming discipline used for every sibling entrypoint including deferred ones; add a required idempotency/call-id argument and specify de-dupe-or-refuse-on-repeat behavior, closing the over-reporting gap the honest-gap register currently only half-covers.

## Hole 3 — AD-4's amendment conflates two different write paths under one trust mechanism, and neither implementer is wrong

AD-4's original mechanism gated **artifact content submission** on a session-scoped, role-matching, single-use engagement token. The v1-driver amendment replaces this with: "role attribution comes from the harness self-reporting `role` on each `llm_call` via AD-26's reporting entrypoint, trusted rather than mechanically gated by a token check."

But AD-26(b)'s reporting entrypoint is about **token/cost telemetry after a model call** — it is not the content-write path at all. AD-4's original gate sat on whatever entrypoint accepts drafted section content (e.g. a `kagami write`-shaped call); AD-26's entrypoint is a distinct, later call reporting `role`, `operation_class`, `model_tier`, tokens, cache-hit for the *ledger*. The amendment's sentence reads as if these are the same gate, but nothing links them:

- **Implementer A** reads the amendment as: content-write entrypoints now take a bare `--role` argument (unauthenticated, since the session/token machinery that would have validated it is exactly what's deferred), independent of any `llm_call` report. This reproduces, by design this time, the honor-system problem the prior full-spine review flagged and closed in its Hole 9 ("Any caller... passes `--role scout` ... and the refusal is satisfied") — except now it is the *sanctioned* interim state, not a bug.
- **Implementer B** reads the amendment literally: there is no `--role` flag on writes at all; instead, attribution for a write is *inferred* by correlating it with the most recent `llm_call` report's `role` field for that turn. This requires a turn/session correlation mechanism that does not exist in the v1-driver state (no session table, no token) — and is ambiguous the moment two roles' calls interleave in one turn (e.g., a Skeptic plausibility check's `llm_call` lands between a Scout search call and Scout's draft write — which prior `llm_call` does the write's attribution pick? nearest-by-time? same-operation-class? unspecified).

A and B are different systems with different spoofability and different data models, and both are defensible readings of one sentence. Neither is ruled out by AD-4, AD-26, or AD-11's new honest-gap bullet ("role attribution is harness-self-reported and trusted, not mechanically token-gated, for v1-driver"), which describes the *weakening* but not the *mechanism*.

This also means the amendment's own "Revisit trigger" — "a role writing off-charter content that a token check would have caught, surfaced during the Epic 7 toy run" — is not something the toy run can reliably surface, because whether such a write is even *detectable* as off-charter depends on which of A/B got built. Under B with a broken correlation heuristic, an off-charter write could be silently mis-attributed to the wrong role rather than flagged at all.

**Close with:** state explicitly which write path carries the `role` claim for content-authorship purposes — most likely: content-write entrypoints take an explicit `--role` argument (self-reported, unauthenticated, exactly as the amendment's honest-gap admission implies), and the `llm_call` report is a *separate*, independently-self-reported signal used only for the token ledger, with no attempted cross-correlation between the two in v1-driver. If cross-correlation is intended instead, specify the join key.

## Hole 4 — AD-27's transcript check and toy-run rubric are not specified precisely enough for two reviewers to agree

**Recorded-transcript check (a).** "A real Claude Code session transcript demonstrating the story's acceptance criteria end-to-end" has no stated bar for *what* the transcript must show beyond "the AC end-to-end." Two PR reviewers can honestly split: Reviewer 1 accepts a clean happy-path transcript (no refusals, no retries) as sufficient if the story's AC don't explicitly demand a negative case; Reviewer 2 holds that "end-to-end" for a story whose AC includes AD-26's ceiling or AD-9's window refusal *requires* the transcript to show the refusal actually firing, not merely that it wasn't triggered by accident. AD-27 gives no minimum-coverage rule (e.g., "one transcript excerpt per AC bullet, including negative-path bullets") the way AD-25 gives the core suite a concrete fixture list ("windows, guards, staleness, E6 ordering, scan-before-write, lease/crash recovery").

**Golden toy-run protocol (c).** "A fixed, repeatable toy investigation (topic, corpus scope, config pinned) whose event log is inspected against the evaluation rubric fixed at Epic 7 story-design time" leaves two things unowned:
- **The rubric's shape and governance.** Is the rubric a versioned artifact (subject to AD-6/AD-23 discipline like everything else the spine cares about), a `schemas/`-registered type, or an ad hoc checklist pasted into a story doc or PR description? AD-14's schema registry enumerates eleven artifact types plus named infrastructure stores (ledger, manifest, event families) — "evaluation rubric" is in none of them. If two different Epic 7 stories (the walking-skeleton story and a later toy-run story) each invent their own rubric format because nothing says otherwise, there is no shared schema a third party could validate a rubric against, and no migration path (AD-23) if the rubric format needs to change later.
- **"Repeatable" against live provider APIs.** AD-7's own Stack entry documents OpenAlex's Feb-2026 pricing/key change and arXiv's tightening 429s as *live, moving facts*. A "fixed... corpus scope" toy run that hits real OpenAlex/S2/arXiv/GitHub endpoints will return different result sets run-to-run as papers are published, retracted, or as rate limits bite differently — undermining "golden" as a regression baseline. AD-27 doesn't say whether the toy run's provider responses are recorded/mocked for true determinism or re-fetched live each time. One implementer builds VCR-style fixture recording (true repeatability, extra plumbing); another just reruns against live APIs and accepts drift (compliant with the letter, but two runs of the "same" golden toy run six months apart are not comparable, silently defeating the mechanism's stated purpose).

**Close with:** for (a), require the transcript to cover every AC bullet including negative/refusal paths where the AC implies one exists, mirroring AD-25's concrete fixture-list precedent. For (c), register the rubric as a lightweight versioned artifact (even a single `schemas/`-governed type suffices) and mandate provider-response recording/replay for the toy run's corpus fetch, or explicitly accept live-API drift as an acknowledged limitation the same way AD-27 already acknowledges its own weaker-than-AD-25 status.

## Hole 5 — AD-11 claims "the one sanctioned read exception"; AD-26(c) quietly adds a second

AD-11's Rule states: "The one sanctioned read exception (FR-5 gate loosening) consumes only `kagami metrics` derived aggregates, never raw events, effective only on researcher approval." That is a closed, singular claim, tied to a specific FR and gated on researcher approval.

AD-26(c) introduces a second consumer of the same `kagami metrics` derived-aggregate surface: "`kagami metrics`'s decision block (FR-37) compares the run's cumulative token ledger against a `config.yaml` soft limit and adds a warning field to the block" — this is unconditional (runs on every `kagami metrics` call, not gated on researcher approval) and serves a different purpose (budget warning / O6 evidence accumulation, not FR-5's gate loosening).

AD-11's text was never edited to reflect this. It still reads "the one," which is now false the moment AD-26 ships — a direct textual inconsistency between two ADs in the same document, of exactly the kind AD-4 and AD-11 were otherwise careful to resolve with dated "v1-driver amendment" / "honest-gap register" notes. A reviewer checking "does anything besides the sanctioned FR-5 exception read `events.jsonl`-derived data" would get different answers depending on whether they read AD-11 in isolation (no) or cross-referenced AD-26 (yes, and it doesn't even require researcher approval).

**Close with:** amend AD-11's Rule to enumerate both sanctioned exceptions (FR-5 gate-loosening, approval-gated; FR-37 budget-warning, unconditional) or generalize the clause to "sanctioned read exceptions are enumerated here" and list them, matching the amendment discipline already used for AD-4 and the honest-gap register itself.

## Hole 6 — AD-26(b)'s "harness calls the entrypoint immediately after every model invocation" is enforcement-by-harness-honor, unreconciled with AD-1's unqualified, unamended text

AD-26's own Prevents clause states the ceiling *must* live in the core rather than skill prose "per AD-1" — reasoning correctly that AD-1 forbids rules existing only in a SKILL.md. But AD-26(b)'s reporting *obligation* — that the harness calls the reporting entrypoint immediately after every model call — is not mechanically enforced by anything in the core; nothing can stop a model call from happening without a following report (the honest-gap register concedes exactly this: "a call made but never reported is invisible to the ledger and audit"). The discipline "call the reporting entrypoint after every model invocation" is, by construction, a rule that exists only in harness prompt/skill logic — precisely the shape AD-1 forbids, "no rule may exist only in a SKILL.md," stated with no qualifier and no amendment note.

AD-4 and AD-11 both received explicit, dated "v1-driver amendment" / new honest-gap-register-entry treatment when the spine chose to weaken them. AD-1 — the AD most directly in tension with AD-26(b)'s honor-system reporting discipline — received no corresponding note, and still reads as an absolute rule ("no rule may exist only in a SKILL.md," `[ADOPTED]`, untouched). A reviewer citing AD-1 verbatim against an Epic 7 story that implements only the honor-system reporting call (no core-side enforcement of the call itself, only of what happens once a report *does* arrive) has a legitimate objection the spine gives no textual way to resolve — is this an accepted, logged exception to AD-1 (like AD-4/AD-11's roles content) or a genuine unresolved contradiction?

**Close with:** add a short amendment clause to AD-1 itself, cross-referencing AD-26(b) and the honest-gap register, acknowledging that the reporting *call itself* is harness-honor (detect via absence, not preventable), the same way AD-4 and AD-11 were amended — closing the asymmetry rather than leaving AD-1 as the one un-amended AD that the new material actually cuts against.

## Hole 7 — `requires_researcher` (AD-26a) doesn't say whether it routes through AD-8's ASK scheduler or AD-14's locked ledger-entry commitment

AD-8: "`kagami ask` is the only entrypoint that creates researcher-facing questions... `kagami ask emit` refuses a batch over 5 and requires target + leverage class per question." AD-14 locks, "from run 1," a stable question-class key in *every* ledger entry. AD-26(a)'s ceiling-exceeded outcome — `requires_researcher` — is a distinct JSON-echo status returned directly by the *original* refused entrypoint (e.g., `kagami write`), not by `kagami ask`. AD-26 never says whether hitting the ceiling internally routes through `kagami ask emit` (getting a question-class key, counting against the ≤5 batch cap, landing in the Question Ledger) or is a separate, ASK-scheduler-bypassing surfacing channel.

Two implementations diverge concretely: **Implementer A** treats `requires_researcher` as a trigger that internally calls `kagami ask emit` with a synthesized leverage class, so it's ledger-recorded and batch-capped like everything else researcher-facing. **Implementer B** treats it as a bare stdout signal the harness must display out-of-band (since AD-26 frames it as "a stop-and-surface outcome," language distinct from AD-8's "researcher-facing questions"), never touching the ledger. Under B, if parallel Deepen workers (AD-10) each trip a ceiling on different sections in the same window, the researcher can be surfaced more than 5 simultaneous stop-and-surface outcomes with no batching — the exact failure AD-8 exists to prevent — and none of them carry a question-class key, silently breaking AD-14's "in every ledger entry" claim (these entries don't exist at all) without AD-14 itself being amended to exempt them.

**Close with:** state explicitly whether `requires_researcher` is implemented as an internal `kagami ask emit` call (recommended — reuses AD-8's batching and AD-14's key commitment for free) or is a genuinely separate channel, and if separate, add an explicit AD-14 exemption note the way AD-14 already carries the constitutive-triad and E6-question exemptions.
